# Custom View Counter Package Plan

## Overview
A lightweight, framework-agnostic npm package for tracking page views natively without inflating counts on user refresh. It will feature client-side deduplication and support modern frameworks like Astro, Next.js, React, and React Native.

## Core Features
- **Smart Deduplication:** Uses browser `localStorage` (or `AsyncStorage` in React Native) to store a timestamped record of visited pages. This prevents duplicate API increment requests if the user refreshes within a set cooldown period (e.g., 24 hours).
- **Framework Agnostic:** Designed to work with plain JavaScript, Astro, React, Next.js, and React Native.
- **Bring Your Own Database (BYOD):** The backend utility will provide simple adapter functions to easily connect to databases like Upstash Redis, Supabase, or any custom endpoint.
- **Real-Time Data:** Fetches the actual number from the database so you can style the counter natively with your own UI/UX components (no ugly forced SVG badges).

## Architecture

### 1. The Client Component (Frontend)
The core of the package will be a vanilla JavaScript utility, along with framework-specific wrappers.
- **Logic:** 
  1. Check if `visited:{slug}` exists in local storage.
  2. If it exists and the timestamp is within the cooldown period, make a `GET` request just to fetch the current count (do not increment).
  3. If it doesn't exist or has expired, make a `POST` request to increment the count, then save the new timestamp in local storage.
- **Planned Exports:**
  - `trackView(slug, apiEndpoint, options)` (Vanilla JS)
  - `useViewCounter(slug, apiEndpoint)` (React Hook for Next.js/React Native)

### 2. The Backend Adapter (Server-Side)
A set of helper functions to handle the server-side logic so developers don't have to rewrite the database transactions.
- **Logic:** Safely connect to the database, increment the sorted set or key, and return the updated count.
- **Planned Exports:**
  - `createRedisAdapter({ url, token })`
  - `increment(slug)`
  - `getViews(slug)`

## How it will be used (Consumer Side)

Once published to npm (`npm i @gohitx/views`), a developer will implement it in two steps:

**1. Create the API Route (e.g., in Astro or Next.js):**
```ts
// src/pages/api/views/[slug].ts
import { createRedisAdapter } from '@gohitx/views/server';

const db = createRedisAdapter({ url: '...', token: '...' });

export const POST = async ({ params }) => {
  const count = await db.increment(params.slug);
  return new Response(JSON.stringify({ count }));
}
```

**2. Use it in the UI:**
```astro
---
// Astro Example
---
<span id="view-counter">Loading...</span>

<script>
  import { trackView } from '@gohitx/views/client';
  
  const count = await trackView('my-blog-post', '/api/views/my-blog-post');
  document.getElementById('view-counter').innerText = count;
</script>
```

## Implementation Phases

### Phase 1: Core Logic & Vanilla JS
- Build the deduplication logic using standard Web Storage APIs.
- Build the backend Redis adapter using `@upstash/redis` as the default fast solution.

### Phase 2: React & Framework Wrappers
- Create a React hook (`useViewCounter`) that handles the `useEffect` lifecycle safely (crucial for preventing double-firing in React 18 strict mode).
- Create a specialized adapter for React Native that injects `AsyncStorage` instead of `window.localStorage`.

### Phase 3: Packaging & Publishing
- Configure a bundler (like `tsup` or `rollup`) to compile the package into both ESM and CJS formats.
- Write a comprehensive `README.md` with copy-paste examples for Astro, Next.js, and React Native.
- Publish to NPM.
