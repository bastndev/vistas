# gohitx-views — Plan Final (simple)

> Basado en PLAN-1, recortado a lo esencial + ranking por página.
> Nada de monorepo, nada de middleware, nada de bot filtering todavía.
> Eso lo agregás *si algún día* lo necesitás, no antes.

---

## Objetivo

Un contador de vistas que:

1. Cuenta 1 vista real por persona (con cooldown, sin inflar con refresh)
2. Funciona en cualquier ruta: `/home`, `/f1`, `/f2`, etc.
3. Te deja ver el número en pantalla (personalizable, sin UI forzada)
4. Te deja ver el **ranking de todas las páginas** en un solo request

Nada más. Sin frameworks múltiples, sin adapters intercambiables, sin
publicar a npm (al menos no en la v1) — eso es opcional para después.

---

## Stack elegido

- **Astro** (ya lo tenés)
- **Upstash Redis** (free tier, REST API, cero servidor que mantener)
- **localStorage** para el cooldown del lado cliente

---

## `src/lib/views.ts` — Lógica central

```ts
import { Redis } from "@upstash/redis";

const redis = new Redis({
  url: import.meta.env.UPSTASH_REDIS_REST_URL,
  token: import.meta.env.UPSTASH_REDIS_REST_TOKEN,
});

const RANKING_KEY = "views:ranking";

export async function incrementView(slug: string) {
  const count = await redis.zincrby(RANKING_KEY, 1, slug);
  return count;
}

export async function getView(slug: string) {
  const score = await redis.zscore(RANKING_KEY, slug);
  return score ?? 0;
}

export async function getRanking() {
  // devuelve [{slug, count}] ordenado de mayor a menor
  const raw = await redis.zrange(RANKING_KEY, 0, -1, {
    rev: true,
    withScores: true,
  });

  const result: { slug: string; count: number }[] = [];
  for (let i = 0; i < raw.length; i += 2) {
    result.push({ slug: raw[i] as string, count: Number(raw[i + 1]) });
  }
  return result;
}
```

**Por qué sorted set y no keys sueltas:** con `zincrby` tenés el
incremento Y el ranking en la misma estructura. Un solo `GET` al final
te da todas tus páginas ordenadas por tráfico — exactamente lo que
pediste ("saber cuál la está rompiendo").

## `src/pages/api/views/[...slug].ts` — Endpoint por página

```ts
import type { APIRoute } from "astro";
import { incrementView, getView } from "../../../lib/views";

export const POST: APIRoute = async ({ params }) => {
  const slug = params.slug ?? "unknown";
  const count = await incrementView(slug);
  return new Response(JSON.stringify({ slug, count }), {
    headers: { "Content-Type": "application/json" },
  });
};

export const GET: APIRoute = async ({ params }) => {
  const slug = params.slug ?? "unknown";
  const count = await getView(slug);
  return new Response(JSON.stringify({ slug, count }), {
    headers: { "Content-Type": "application/json" },
  });
};
```

> Usá `[...slug].ts` (rest param), no `[slug].ts`. Así también soporta
> rutas anidadas como `extension/lynx-theme-pro` si las necesitás después.

## `src/pages/api/views/ranking.ts` — Para vos, el dashboard

```ts
import type { APIRoute } from "astro";
import { getRanking } from "../../../lib/views";

export const GET: APIRoute = async () => {
  const ranking = await getRanking();
  return new Response(JSON.stringify(ranking), {
    headers: { "Content-Type": "application/json" },
  });
};
```

Pegás esa URL en el navegador o la consumís en una página privada tipo
`/admin/stats` y ya tenés tu propio mini-dashboard de cuál ventana se
está rompiendo de tráfico.

## `src/components/ViewCounter.astro` — Cliente con dedupe

```astro
---
interface Props {
  slug: string;
  cooldownHours?: number;
}
const { slug, cooldownHours = 24 } = Astro.props;
---
<span class="view-counter" data-slug={slug} data-cooldown={cooldownHours}>
  —
</span>

<script>
  document.querySelectorAll<HTMLElement>(".view-counter").forEach(async (el) => {
    const slug = el.dataset.slug!;
    const cooldownHours = Number(el.dataset.cooldown);
    const storageKey = `viewed:${slug}`;
    const last = localStorage.getItem(storageKey);
    const cooldownMs = cooldownHours * 60 * 60 * 1000;

    const isFresh = last && Date.now() - Number(last) < cooldownMs;
    const method = isFresh ? "GET" : "POST";

    const res = await fetch(`/api/views/${slug}`, { method });
    const data = await res.json();

    el.textContent = data.count.toString();

    if (!isFresh) {
      localStorage.setItem(storageKey, Date.now().toString());
    }
  });
</script>
```

Esto ya resuelve "personalizable": el componente no impone estilos, vos
le pones el CSS que quieras al `<span class="view-counter">`.

## Fases (realistas, no 5 fases con "futuro" indefinido)

### Fase 1 — Esto de arriba, funcionando
- `lib/views.ts` + 2 endpoints + componente
- Probado en `home`, `f1`, `f2`

### Fase 2 — Dashboard simple para vos
- Página `/admin/stats.astro` que llama a `/api/views/ranking`
- Tabla ordenada: página → vistas

### Fase 3 (opcional, solo si lo necesitás)
- Extraer a paquete npm propio si querés reusarlo en otro proyecto
- Bot filtering básico (ignorar user-agents de crawlers conocidos)

**No hay Fase 4 ni 5.** Si en el futuro necesitás Vue, Svelte, React
Native, etc., los agregás cuando realmente tengas ese proyecto — no
antes, porque hoy no los necesitás y mantenerlos sin uso real es
trabajo muerto.

## Qué saqué del PLAN-2 (ViewPulse) y por qué

| Feature de ViewPulse | Por qué no lo metí ahora |
|---|---|
| Monorepo con 8 paquetes | Para un solo consumidor (vos) es overhead puro |
| Soporte React/Vue/Svelte/Next/Native | No los usás hoy; Astro te alcanza |
| Middleware system (`botFilter`, `logger`, `geo`) | Nadie lo necesita en v1; lo armás si surge el problema real |
| Bot filtering | Bueno a futuro, pero no bloquea tu caso de uso actual |
| SPA auto-tracking (`pushState`/`popstate`) | Astro no es SPA por default; no aplica |
| Múltiples adapters de DB intercambiables | Vos ya elegiste Redis/Upstash, no necesitás abstraer eso todavía |

Todo eso es válido **si algún día publicás esto como librería pública**
para que otros devs la usen en sus propios stacks. Pero hoy tu problema
es "contar visitas en mi portfolio personal", y ese problema se resuelve
con ~80 líneas de código, no con una arquitectura de plugin system.
